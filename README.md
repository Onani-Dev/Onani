# Onani

A booru-style imageboard built with **Flask** (API) and **Vue 3** (SPA frontend).

## Releases

Releases are tag-driven via GitHub Actions.

- Beta releases use tags like `v1.0.0-beta.1`
- Stable releases use tags like `v1.0.0`
- Pushing either tag creates a GitHub Release automatically
- Beta tags are published as GitHub prereleases
- Tagged releases also publish production container images to GHCR
- CI builds only the production `flask`, `celery`, and `frontend` images
- The optional in-repo `nginx` image is not part of CI builds

Current frontend version metadata lives in `frontend/package.json`, and the SPA
footer links to the matching GitHub release page for that version.

## Architecture

```
Onani/
├── services/       ← Pure business logic (no Flask globals)
├── controllers/    ← Route decorators + backward-compat shim
├── models/         ← SQLAlchemy models + Marshmallow schemas
├── routes/
│   ├── api/v1/     ← REST API (Flask-RESTful)
│   ├── auth/       ← Legacy server-rendered auth
│   ├── views/      ← Legacy Jinja2 views
│   └── spa.py      ← SPA catch-all (/)
├── importers/      ← gallery-dl importer (900+ sites)
├── tasks/          ← Celery async tasks
└── cron/           ← Scheduled jobs

frontend/           ← Vue 3 + Vite SPA
```

### Service layer (`Onani/services/`)

All database and business logic lives here as plain Python — no Flask
request/response, `current_user`, `flash()`, or `abort()` calls. Services
accept explicit parameters (the acting user, config values) and raise typed
exceptions (`BanError`, `InvalidCredentialsError`, etc.) that route handlers
catch and translate into HTTP responses.

Authentication uses Flask-Login session cookies. The SPA bootstraps a CSRF
token from `/auth/csrf` and sends it on mutating requests via the
`X-CSRFToken` header.

### Frontend (`frontend/`)

Vue 3 SPA served at the root (`/`). Built with:

- **Vite** — build tooling
- **Vue Router 4** — client-side routing (HTML5 history mode)
- **Pinia** — state management (auth store)
- **Axios** — HTTP client with CSRF interceptor

In development, Vite's dev server proxies `/api` and `/static` to Flask on
port 5000.

## Deployment Model

Onani does not require an in-repo Nginx container.

- Production app server: Flask + Gunicorn on port 5000.
- Edge proxy/TLS: provided by your existing reverse proxy in front of Gunicorn.
- Media and thumbnails: generated by Flask services, stored on disk in sharded
  directories.

## Modes

The project has two primary run modes:

1. **`prod` mode**: Flask + Gunicorn (production settings), plus DB/Redis/Celery.
2. **`dev` mode**: Flask debug server + Vite dev server (hot reload), plus DB/Redis/Celery.

## Running (Start to Finish)

### Prerequisites

You need **one** of the following:

- **Podman** + **podman-compose** — primary container workflow for this repo
- **Nix** — `flake.nix` provides a complete dev shell
- **Manual** — Python 3.10+, Node.js 22+, PostgreSQL 14, Redis 7, ffmpeg

### Podman setup

Onani is designed around `podman-compose`.

Minimum tools:

- `podman`
- `podman-compose`

Verify your install:

```bash
podman --version
podman-compose --version
podman info
```

Optional shell aliases if you habitually type Docker commands:

```bash
alias docker=podman
alias docker-compose=podman-compose
```

Useful host path configuration in `.env`:

- `IMAGES_HOST_DIR` — where uploaded post media is stored on the host
- `AVATARS_HOST_DIR` — where avatars are stored on the host
- `DATABASE_HOST_DIR` — where PostgreSQL data is stored on the host
- `DEEPDANBOORU_MODEL_HOST_DIR` — local path for the optional DeepDanbooru model directory

If these are unset, `docker-compose.yml` falls back to repo-local directories such as
`images`, `avatars`, `database`, and `./models/deepdanbooru`.

### 1. Generate environment file

```bash
python generate_env.py
```

The script generates a `.env` with `DB_PASSWORD` and `FLASK_SECRET_KEY`.

### 2a. Podman — prod mode

```bash
podman-compose -f docker-compose.yml --profile prod up -d --build
```

This starts the production-mode app stack.

The Flask entrypoint automatically runs:

- `flask db init`
- `flask init-db`
- `flask db upgrade`

Then Gunicorn starts on `0.0.0.0:5000`.

Once healthy:

| URL | What |
| --- | ---- |
| http://flask:5000/ (container network) | Flask app endpoint (SPA catch-all + API mount) |
| http://flask:5000/api/v1/ (container network) | REST API (Gunicorn direct) |

In `prod` profile, `flask` is **not published to the host** by default
(`expose` only). Your reverse proxy should connect to it on the container/
private network. Terminate TLS and set forwarded headers at the external proxy.

Example proxy behavior:

- `/api/*` -> `http://onani-flask:5000/api/*`
- `/static/*` -> `http://onani-flask:5000/static/*`
- all other routes -> `http://onani-flask:5000/` (SPA catch-all)

Build the frontend bundle for production (if you are serving the SPA via Flask):

```bash
cd frontend
npm ci
npm run build
```

Then place/copy the resulting `dist` assets where the Flask SPA blueprint can
serve them.

### 2a.1 Production Nginx reverse proxy example

If your public entrypoint is Nginx, put it in front of Gunicorn and keep
Gunicorn private (for example only reachable on an internal network).

```nginx
upstream onani_app {
  server 127.0.0.1:5000;
}

server {
  listen 80;
  server_name your-domain.example;

  # If you host Vue dist with nginx directly, point this at your built frontend dist.
  root /srv/onani/frontend/dist;
  index index.html;

  # Long-cache hashed Vue assets
  location /assets/ {
    try_files $uri =404;
    expires 1y;
    add_header Cache-Control "public, immutable";
  }

  # API
  location /api/ {
    proxy_pass http://onani_app;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
  }

  # Media routes served by Flask (raw files + generated thumbnails/sample)
  location ~ ^/(images|avatars|sample)/ {
    proxy_pass http://onani_app;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
  }

  # Static files from Flask app package (robots.txt, manifest, etc.)
  location /static/ {
    proxy_pass http://onani_app;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
  }

  # SPA fallback
  location / {
    try_files $uri $uri/ /index.html;
  }
}
```

If you prefer Flask to serve the SPA index and assets, proxy `/` to Gunicorn
instead of using `root` + `try_files`.

Seed default tags:

```bash
podman-compose -f docker-compose.yml --profile prod exec flask flask tags --filename meta.json
podman-compose -f docker-compose.yml --profile prod exec flask flask tags --filename explicit.json
```

Create an admin user:

```bash
podman-compose -f docker-compose.yml --profile prod exec flask flask add-owner --id <USER_ID>
```

### 2b. Podman — development

```bash
./build.sh dev
```

Equivalent explicit command:

```bash
podman-compose -f docker-compose.yml --profile dev up -d --build
```

Same as production, but:

- Source code is bind-mounted (`./:/onani`)
- Flask runs in debug mode (`flask run --debug`)
- Flask is exposed directly on **:5000**
- PostgreSQL on **:5432** and Redis on **:6379** are exposed to the host
- `FLASK_ENV=development`, `TESTING=True`, SQL echo on
- Cron jobs are started (`crond` + `flask crontab add`)
- `frontend-dev` runs Vite on **:5173**

To work on the Vue SPA during development, run Vite separately:

```bash
cd frontend
npm install
npm run dev          # starts on :5173, proxies /api + /static → Flask :5000
```

| URL | What |
| --- | ---- |
| http://localhost:5173/ | Vue SPA (Vite dev server, hot reload) |
| http://localhost:5000/api/v1/ | Flask API direct |
| http://localhost:5000/ | Flask catch-all (if SPA build exists) |

### 2c. Nix — fully local (no Docker)

```bash
nix develop                       # enters dev shell with Python, Node, ffmpeg, etc.
pip install -r requirements.txt   # Python deps into .venv (auto-created)
```

You need PostgreSQL and Redis running locally (or use Docker just for those):

```bash
# Option A: start just Postgres + Redis via Podman
podman-compose -f docker-compose.yml up -d postgres redis

# Option B: use system services
# Make sure DATABASE_URL points to your local Postgres
```

Then start the backend and frontend in separate terminals:

```bash
# Terminal 1 — Flask dev mode
export DATABASE_URL=postgresql://onani_db:onani_db@localhost:5432/onani_db
flask run --debug                 # starts on :5000

# Terminal 2 — Vue dev mode
cd frontend && npm install && npm run dev   # starts on :5173
```

For manual production-style local run:

```bash
# Terminal 1 — Flask via Gunicorn
export DATABASE_URL=postgresql://onani_db:onani_db@localhost:5432/onani_db
gunicorn -b 0.0.0.0:5000 -w 10 --threads 100 run:app
```

## CLI Commands

Run via `flask <command>` or through compose:

- Prod stack: `podman-compose -f docker-compose.yml --profile prod exec flask flask <command>`
- Dev stack: `podman-compose -f docker-compose.yml --profile dev exec flask-dev flask <command>`

Common commands:

| Command                          | Description                     |
| -------------------------------- | ------------------------------- |
| `init-db`                        | Create database tables          |
| `drop-db`                        | Drop and recreate tables (testing only) |
| `tags --filename FILE`           | Seed tags from a defaults JSON  |
| `add-owner --id ID`              | Promote user to owner role      |
| `reset-password --id ID`         | Reset a user's password         |
| `disable-otp --id ID`            | Disable 2FA for a user          |
| `db init`                        | Initialize migrations directory |
| `db migrate`                     | Generate Alembic migration      |
| `db upgrade`                     | Apply pending migrations        |

## Media Storage and Thumbnails

Uploaded files are stored on disk using a two-level sharded layout handled by
Flask services.

- Path strategy: first two chars of filename are used as shard directory.
- Example: `abcdef1234.jpg` -> `IMAGES_DIR/ab/abcdef1234.jpg`
- Helper functions: `shard_path()` and `ensure_shard_dir()` in
  `Onani/services/files.py`

Video thumbnails are generated by Flask (ffmpeg-python), not by Nginx.

- On upload: a JPEG thumbnail is generated for video files.
- Backfill task: admin task `backfill_video_thumbnails` generates missing
  thumbnails.
- Thumbnail filename: `<video_stem>.jpg` in the same sharded layout.

## Container Services

| Service    | Ports (dev) | Purpose                           |
| ---------- | ----------- | --------------------------------- |
| `flask`    | 5000        | API + SPA catch-all (gunicorn)    |
| `postgres` | 5432        | PostgreSQL 14                     |
| `redis`    | 6379        | Cache + Celery broker             |
| `celery`   | —           | Async task workers (10 concurrent)|
| `frontend-dev` | 5173    | Vite dev server (optional)        |

Both modes use the same `docker-compose.yml`. Postgres and Redis are shared;
Flask and Celery have profile-specific variants. In-repo Nginx services are
optional and not required for the app to run.

Podman profile summary:

- `prod` — `flask`, `celery`, `postgres`, `redis`
- `dev` — `flask-dev`, `frontend-dev`, `celery-dev`, `postgres`, `redis`
- `proxy` — optional in-repo `nginx`/`nginx-dev` for local reverse-proxy testing only
- `frontend-build` — one-off frontend bundle build container

Optional extra profiles:

- `proxy`: starts in-repo Nginx services (legacy/testing only)
- `frontend-build`: builds frontend dist artifact container
